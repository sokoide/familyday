### 構成
- **バックエンド**: `2026/server/`(Go・Clean Architecture 4層)。`domain`→`usecase`→`adapters`(presentation/infra)→`app`(ワイヤリング)→`cmd/server`。
- **フロント**: `2026/web/`(TypeScript + Vite・バニラ)。`app`(状態機械) / `ports` / `infra`(fetch・WebSpeech) / `ui`(qr/share)。日/英 i18n(`messages.ts`)。
- **Makefile**: `2026/Makefile`(`cd 2026 && make ...`)。build/unit/integration/vet/fmt/check/run/dev/help。

### 今回(2fe58b4)の変更: Gemini 3.1系への更新 + 画像生成API移行
- **モデル**: judge/story=`gemini-3.1-flash-lite`、画像=`gemini-3.1-flash-lite-image`(Nano Banana Lite)。
- **画像API移行**: `GenerateImages`(Imagen系・**2026-08-17廃止**)→`GenerateContent`へ。レスポンスの `InlineData`(`Blob`)から画像取り出し(`firstInlineImage`)。
- **リネーム**: `Config.ModelImagen`→`ModelImage`、`ImagenGenerator`→`ImageGenerator`、env `GEMINI_MODEL_IMAGEN`→`GEMINI_MODEL_IMAGE`。
- **画像サイズ/枚数のConfig化**:
  - `ImageSize`(既定**1024**=モデルのネイティブ出力・リサイズなし)。1024以外は生成後にサーバ側でリサイズ(`golang.org/x/image/draw` CatmullRom)。他モデル移行時や縮小用途。
  - `ImageCount`(既定**1**=非バッチ)。>1 は複数回生成して最初の成功を採用(コストN倍)。
  - env: `GEMINI_IMAGE_SIZE` / `GEMINI_IMAGE_COUNT`。README/.env.example/コードコメントに設定方法を明記。
  - `resizeSquare` の単体テスト追加(ネットワーク不要・緑)。
- **診断ログ**: usecase に `Logger` ポート追加。画像生成失敗(課金/クォータ等)がサーバログに出るようになった(従来は握り潰し)。

## ⚠️ 重要:画像生成は有料APIキーが必須(未検証)

- **現状**: 無料ティアの Gemini API キーでは `gemini-3.1-flash-lite-image` の quota が **0**(`429 RESOURCE_EXHAUSTED / limit: 0`)で**画像生成が失敗する**ことを実APIで確認済み。
- 判定/ストーリー(`gemini-3.1-flash-lite`)は無料ティアでも動作した。
- **ユーザーが有料APIキーを後日作成予定**。それまでは画像生成の実API検証ができない。
- 画像生成失敗時はフロント `onerror` で data-URI fallback 絵文字画像にフォールバックするので体験は止まらない。

## 次回の具体的ステップ

### 1. 有料APIキーで画像生成を検証(最優先)
1. 有料(Pay-as-you-go)の Gemini API キーを作成し `2026/.env` の `GEMINI_API_KEY` に設定。
2. `make integration` を実行。`TestIntegration_ImageDirect`(infra 直接呼び出し)と `TestIntegration_Ending`/`TestIntegration_ImageFileWritten` が成功すること。`TestIntegration_ImageFileWritten` は画像ファイル未書き出し時に skip するが、有料キーなら書き出されるはず(必要なら2秒のポーリング期限を延ばす)。
3. `curl -X POST localhost:8080/api/ending ...` で `imageUrl` が 200 で画像を返すことを確認。
4. リサイズ動作確認: `GEMINI_IMAGE_SIZE=512` で起動し、生成画像が 512x512 になることを確認。

### 2. 音声入力(macOS Edge / 本番 Windows)の確定
- macOS Edge で `[speech] error:` のコードを確認(コンソール)。権限/ネットワーク/audio-capture 等。
- 会場は Windows 11 Edge。**Windows 実機でのマイク動作確認が必須**(macOS の失敗≠本番失敗)。
- 不安定ならサーバサイドSTT(Gemini/Whisper)への移行を検討。

### 3. 本番デプロイ
- 公開Linuxサーバへ `2026/` 配置 → `make build` / `make build-bin`。
- `2026/.env` の `PUBLIC_BASE_URL=https://你的域名` 設定。
- nginx 等で HTTPS → `:8080`。systemd で `2026/server/bin/familyday` 常駐。

### 4. その他
- 判定プロンプトの実機チューニング(`domain/catalog.go` / `gemini/prompts.go`)。
- 結果ページ `/r/{id}` の言語切替ボタン追加(任意)。

## よく使うコマンド

```bash
cd 2026
make                  # build + unit(既定)
make integration      # 実APIのE2E(GEMINI_API_KEY 必要・画像は有料キーが必要)
make run              # サーバ起動(2026/.env を読む)
make check            # Clean Architecture チェック
```

## 設計上の判断点(記録)
- ライフ权威はフロント保持(サーバステートレス)。`/api/ending` の cleared はクライアント権威(閉鎖イベントのため許容)。
- Gemini セーフティブロック/空応答は Bad(リトライ)に倒す。真の通信・認証エラーは upstream(502)。
- 画像生成失敗はフォールバック(data-URI)しつつログ出力。
- フロントはバニラ TS(画面3枚・バンドル最小)。`state.ts`/`ports`/`infra` はフレームワーク非依存。
- 来年は `2026/` を `2027/` にコピーし、`domain/catalog.go` と `web/src/app/messages.ts` を差し替えれば再利用可能。
