### 起動時 Gemini キー/モデル健全性チェック追加(2026-07-09)

`make run` 時に「サーバは起動するのに判定APIが毎回130msで502になる」現象を調査。
原因は `genai.NewClient` が APIキーを保持するだけで Google 検証しないため、
無効/期限切れ/割り当て枯渇のキーでもサーバが起動し、初回 `GenerateContent` で
即座に 401/403(~130ms)→ `ErrUpstream` → 502 → フロント netError となること。
`.env` 有無は無関係(godotenv.Load は既存環境変数を上書きしないことを実証済み)。

- **`server/internal/adapters/infra/gemini/client.go`**:
  - `(*Client).CheckHealth(ctx)` を追加。`Models.Get` で `ModelJudge`/`ModelStory`
    の両モデルを検証(課金不要・401/404を即検出・10s タイムアウト・同一モデルは重複回避)。
- **`server/cmd/server/main.go`**:
  - `BuildMux` 前に NewClient→CheckHealth を実行。失敗時 `log.Fatalf` で即終了。
  - 一時的上流障害耐性のため3回リトライ(2s 間隔)。無効キーは即失敗しリトライも即終了するため bad-key 検出は遅れない。
  - `opts.APIKey` を `os.Getenv` 直呼びから `apiKey` 変数に統一。
  - `main.go` に置いたため `BuildMux` を使うユニット/統合テストには影響しない。
- **検証**: `go build` / `go vet` / `go test ./...` / cleanarch check 全て成功。
  有効キー→起動、無効キー→3回リトライ後 fatal、空キー→即 fatal、不正モデル→404 fatal を実機確認。

## 設計上の判断点

- Clean Architecture の依存方向は維持。UseCase は Gemini や HTTP 具象を知らず、Composition Root が adapter を注入する。
- LLM の遅延・失敗はユーザー体験上の通常ケースとして扱い、サーバは context timeout、フロントは AbortController timeout で制御する。
- 乱数ID生成は予測可能なフォールバックを使わず、失敗時はエラーとして返す。
- フロントの非同期処理は `gameEpoch` と `state.phase` で古い結果を破棄する方針。
