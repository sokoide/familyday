### 構成
- **バックエンド**: `2026/server/`(Go・Clean Architecture 4層)

  - QRコード + メール送信(mailto:・アドレスはサーバへ送らない)
- **Makefile**: `2026/Makefile` に集約(`cd 2026 && make ...`)。build/unit/integration/vet/fmt/check/run/dev/help

### 設計上の判断点(記録)
- ライフ权威はフロント保持(サーバはステートレス・3並列でセッション管理不要)。→ `/api/ending` の cleared はクライアント権威なので捏造可能。閉鎖イベントのため許容。

## 進行中・未完了(調査中)

### 音声入力が macOS Edge で失敗する問題
- 現象: macOS Edge でマイクボタンを押すと「えらーがおきました」となり音声が取れない。
- 原因の見立て: Web Speech API はクラウド型(OS権限 + ネットワーク + 音声サービス到達性 が必要)。macOS Edge では `network` / `audio-capture` / OSマイク権限不足 等。現コードは実コードをコンソールに出すよう改善済み。
- 重要: 会場は Windows 11 Edge。Windows は OS 音声サービス統合で macOS より安定。**macOS の失敗 = 本番失敗ではないが、Windows 実機確認は必須**。

## 次回の具体的ステップ

### 音声問題の確定(最優先)
1. macOS Edge で F12›コンソールを開きマイクボタン押下 → `[speech] error: ...` のコードを確認:
   - `speech-not-allowed` / `service-not-allowed` → システム設定›マイク で Edge に権限付与
   - `speech-network` → ネットワーク/プロキシ/ファイアウォール確認
   - `speech-audio-capture` → 他アプリのマイク占有確認・入力デバイス確認
   - `speech-not-supported` → 非セキュアコンテキスト(http://LAN-IP)。localhost or https で開く
2. コードが分かったら対処(権限か、ネットワークか、後述のサーバSTT移行か)を決定。

### 本番前(必須)
3. **Windows 11 Chrome/Edge 実機検証** — 会場同環境でマイク許可・Web Speech API の実動作確認。
4. **本番デプロイ**
   - 公開Linuxサーバへ `2026/` を配置
   - `cd 2026 && make build`、`make build-bin` でバイナリ生成
   - `2026/.env` に実際の `GEMINI_API_KEY` と `PUBLIC_BASE_URL=https://你的域名` を設定
   - nginx 等で HTTPS(独自ドメイン) → `:8080` へリバースプロキシ(マイク利用にHTTPS必須)
   - systemd 等で `2026/server/bin/familyday` を常駐
5. **判定プロンプトの実機チューニング** — `domain/catalog.go` の成功条件と `gemini/prompts.go` を調整。

## よく使うコマンド

```bash
cd 2026
make                  # build + unit(既定)
make integration      # 実APIのE2E(GEMINI_API_KEY 必要・無いとskip)
make run              # サーバ起動(2026/.env を読む)
make dev              # フロント開発サーバ
make check            # Clean Architecture チェック
```

ローカル起動:
```bash
cd 2026
make build            # 初回(フロント成果物は server/static/ へ)
cp .env.example .env  # GEMINI_API_KEY 記入
make run              # http://localhost:8080
```
